# Field Reference — Society Management System

Scope: `x_society` · SDK: `@servicenow/sdk@4.5.0`

All custom table and field names are prefixed with `x_society_`.
"Short name" is what you use in scripts; "Scoped name" is what ServiceNow stores in the dictionary.

---

## Table: x_society_flat

Master flat/unit registry. All other tables hold a reference to this table.
Display field: `flat_number`

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `flat_number` | `x_society_flat_number` | String | max 20, mandatory | Human-readable unit identifier (e.g., "A-101") |
| `block` | `x_society_block` | String | max 40 | Block or tower name within the society |
| `floor` | `x_society_floor` | Integer | — | Floor number where the unit is located |
| `flat_type` | `x_society_flat_type` | Choice | see values | Configuration of the unit (1–4 BHK) |
| `owner_name` | `x_society_owner_name` | String | max 100, mandatory | Full name of the registered owner |
| `owner_phone` | `x_society_owner_phone` | String | max 15 | Owner's mobile number |
| `owner_email` | `x_society_owner_email` | String | max 100 | Owner's email address |
| `tenant_name` | `x_society_tenant_name` | String | max 100 | Full name of the current tenant (if any) |
| `tenant_phone` | `x_society_tenant_phone` | String | max 15 | Tenant's mobile number |
| `occupancy_status` | `x_society_occupancy_status` | Choice | see values, mandatory, default `owner_occupied` | Who currently occupies the unit |
| `area_sqft` | `x_society_area_sqft` | Integer | — | Carpet area in square feet |
| `parking_slot` | `x_society_parking_slot` | String | max 20 | Assigned parking slot identifier |
| `active` | `x_society_active` | Boolean | default `true` | Whether this flat is actively managed in the system |

**flat_type values:** `1bhk` (1 BHK) · `2bhk` (2 BHK) · `3bhk` (3 BHK) · `4bhk` (4 BHK)

**occupancy_status values:** `owner_occupied` (Owner Occupied) · `tenant_occupied` (Tenant Occupied) · `vacant` (Vacant)

## Table: x_society_maintenance_bill

One record per flat per month. Auto-generated by the monthly scheduled job.
Display field: auto-number · Auto-number prefix: `BILL` (e.g., BILL0001000)

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `flat` | `x_society_flat` | Reference → `x_society_flat` | mandatory | The unit this bill belongs to |
| `month_year` | `x_society_month_year` | String | max 7, mandatory | Billing period in `YYYY-MM` format (e.g., `2024-03`) |
| `amount` | `x_society_amount` | Decimal | 2 decimal places, mandatory | Maintenance charge in Indian Rupees |
| `due_date` | `x_society_due_date` | Date | mandatory | Date by which payment must be made |
| `payment_date` | `x_society_payment_date` | Date | — | Actual date on which payment was received |
| `payment_mode` | `x_society_payment_mode` | Choice | see values | How the payment was made |
| `status` | `x_society_status` | Choice | see values, mandatory, default `unpaid` | Current payment state of the bill |
| `notes` | `x_society_notes` | Multi-line Text | max 500 | Free-text notes about this bill or payment |

**payment_mode values:** `cash` (Cash) · `upi` (UPI) · `bank_transfer` (Bank Transfer) · `cheque` (Cheque)

**status values:** `unpaid` (Unpaid) · `paid` (Paid) · `overdue` (Overdue)

## Table: x_society_service_request

Unified table for complaints, facility bookings, and notices. The `request_type` field determines which subset of fields is relevant.
Display field: `short_description` · Auto-number prefix: `SR` (e.g., SR0001000)

### Core fields (all request types)

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `flat` | `x_society_flat` | Reference → `x_society_flat` | mandatory | The unit associated with this request |
| `requested_by` | `x_society_requested_by` | Reference → `sys_user` | — | The user who raised the request |
| `request_type` | `x_society_request_type` | Choice | see values, mandatory | Determines which category this record belongs to |
| `short_description` | `x_society_short_description` | String | max 160, mandatory | One-line summary of the request |
| `description` | `x_society_description` | Multi-line Text | max 4000 | Detailed description or body text |
| `status` | `x_society_status` | Choice | see values, mandatory, default `open` | Current lifecycle state (valid values differ by request_type — see below) |
| `opened_on` | `x_society_opened_on` | DateTime | read-only | Timestamp set automatically by BR `set-dates` on insert |
| `resolved_on` | `x_society_resolved_on` | DateTime | read-only | Timestamp set automatically by BR `set-dates` when resolved |

**request_type values:** `complaint` (Complaint) · `facility_booking` (Facility Booking) · `notice` (Notice)

**status values (all):** `open` · `in_progress` · `resolved` · `rejected` · `requested` · `confirmed` · `cancelled` · `active` · `expired`

### Complaint-only fields

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `priority` | `x_society_priority` | Choice | default `medium` | Urgency level of the complaint |
| `category` | `x_society_category` | Choice | see values | Type of maintenance or issue being reported |
| `assigned_to` | `x_society_assigned_to` | Reference → `sys_user` | — | Staff member or committee member handling the complaint |
| `vendor_name` | `x_society_vendor_name` | String | max 100 | Name of the external vendor engaged for resolution |
| `resolution_notes` | `x_society_resolution_notes` | Multi-line Text | max 2000 | Details of how the complaint was resolved |

**priority values:** `low` (Low) · `medium` (Medium) · `high` (High) · `critical` (Critical)

**category values:** `plumbing` (Plumbing) · `electrical` (Electrical) · `cleaning` (Cleaning) · `security` (Security) · `lift` (Lift) · `other` (Other)

### Facility booking-only fields

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `facility` | `x_society_facility` | Choice | see values | Which amenity is being booked |
| `booking_date` | `x_society_booking_date` | Date | — | Calendar date for the booking |
| `start_time` | `x_society_start_time` | Time | — | Start time of the booking slot |
| `end_time` | `x_society_end_time` | Time | — | End time of the booking slot |
| `charges` | `x_society_charges` | Decimal | 2 decimal places | Usage fee in Indian Rupees (if applicable) |

**facility values:** `clubhouse` (Clubhouse) · `gym` (Gym) · `swimming_pool` (Swimming Pool) · `community_hall` (Community Hall) · `terrace` (Terrace)

### Notice-only fields

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|
| `pinned` | `x_society_pinned` | Boolean | default `false` | Whether the notice is pinned to the top of the portal |
| `expires_on` | `x_society_expires_on` | Date | — | Date on which the notice automatically moves to `expired` status |

| Short name | Scoped name | Type | Constraint | Description |
|---|---|---|---|---|

| `visitor_phone` | `x_society_visitor_phone` | String | max 15 | Visitor's mobile number |
| `visitor_vehicle` | `x_society_visitor_vehicle` | String | max 20 | Vehicle registration number (if arriving by vehicle) |
| `purpose` | `x_society_purpose` | Choice | see values | Reason for the visit |
| `check_in` | `x_society_check_in` | DateTime | mandatory | Date and time the visitor entered the premises |
| `check_out` | `x_society_check_out` | DateTime | — | Date and time the visitor exited (blank until they leave) |
| `entry_by` | `x_society_entry_by` | Reference → `sys_user` | — | Security staff member who logged this entry |

## request_type Field Map

Which fields are relevant for each request type:

### complaint
`request_type = 'complaint'`
Core fields + `priority`, `category`, `assigned_to`, `vendor_name`, `resolution_notes`

### facility_booking
`request_type = 'facility_booking'`
Core fields + `facility`, `booking_date`, `start_time`, `end_time`, `charges`

> The BR `prevent-double-booking` checks `facility + booking_date + start_time/end_time` for conflicts before saving.

### notice
`request_type = 'notice'`
Core fields + `pinned`, `expires_on`

## Status Workflows

```
Maintenance Bill
  unpaid ──► paid        (manual — set by committee/admin on payment)
  unpaid ──► overdue     (auto — daily scheduled job at 1 AM past due_date)

Complaint
  open ──► in_progress ──► resolved
                       └──► rejected

Facility Booking
  requested ──► confirmed
            └──► cancelled

Notice
  active ──► expired     (auto — when current date passes expires_on)
```

## Script Include Public Methods

### SocietyUtils

```javascript
// Check if a facility slot is free. Used by the prevent-double-booking BR.
// Returns true if available, false if conflicting record exists.
SocietyUtils.isSlotAvailable(facility, date, start_time, end_time)  // → boolean

// Returns a GlideRecord of all unpaid bills for a given flat.
// Caller must iterate the returned GlideRecord with .next().
SocietyUtils.getUnpaidBillsByFlat(flat_sys_id)  // → GlideRecord

// Returns a GlideRecord of open service requests filtered by type.
// request_type must be one of: 'complaint', 'facility_booking', 'notice'
SocietyUtils.getOpenRequestsByType(request_type)  // → GlideRecord
```

### MaintenanceBillingEngine

```javascript
// Creates one x_society_maintenance_bill record per active flat for the given month.
// month_year format: 'YYYY-MM' (e.g., '2024-03')
// Called by the monthly scheduled job on the 1st of each month.
MaintenanceBillingEngine.generateMonthlyBills(month_year)  // → void

// Finds all unpaid bills whose due_date is in the past and sets status = 'overdue'.
// Called by the daily scheduled job at 1 AM.
// Returns the count of records updated.
MaintenanceBillingEngine.markOverdueBills()  // → number
```
